# Payments Spec

## Summary

buddy-note에 유료 멤버십, 무료 체험, 정기 결제를 도입한다. v1은 웹/PWA에서 한국 사용자에게 익숙한 카드 정기 결제를 먼저 제공하고, 서버가 신뢰할 수 있는 구독 상태와 entitlement를 기준으로 프리미엄 기능 접근을 제어한다.

## Background

buddy-note의 장기 가치는 반려동물의 기억이 누적되는 데 있다. 무료 사용자는 첫 기록과 기본 일기 생성을 경험하고, 유료 사용자는 더 긴 기억 보존, 프리미엄 테마, 가족 기능, 월간 하이라이트 같은 누적 가치가 큰 기능을 사용할 수 있게 한다.

결제 정책은 제품의 핵심 루프를 막지 않아야 한다. 무료 경험은 “내 강아지의 첫 AI 일기”를 만들 수 있을 만큼 열려 있어야 하고, 유료 경험은 시간이 지날수록 더 강해지는 기억, 공유 이미지, 가족 참여, 하이라이트 편집에 집중한다.

## Goals

- 웹/PWA 기준 유료 멤버십과 정기 결제를 받을 수 있다.
- 신규 사용자가 프리미엄 기능을 부담 없이 체험할 수 있는 무료 체험 정책을 제공한다.
- 유료 기능 접근은 서버에서 entitlement 기준으로 통제한다.
- 결제 성공, 실패, 해지, 환불 이벤트를 추적 가능하게 남긴다.
- 사용자가 현재 멤버십 상태와 다음 결제일, 해지 여부를 이해할 수 있다.
- 운영자가 고객 문의, 결제 실패, 환불 요청을 최소 정보로 확인하고 대응할 수 있다.

## Non-Goals

- v1에서 iOS/Android 네이티브 인앱결제까지 동시에 구현하지 않는다.
- PG사 여러 개를 동시에 붙이지 않는다.
- 복잡한 쿠폰, 프로모션, 팀/비즈니스 요금제는 v1 범위에서 제외한다.
- 관리자용 풀 대시보드는 v1에서 만들지 않는다. 최소 SQL/admin query로 확인 가능한 수준을 우선한다.
- 결제 시스템으로 로그 생성 비용 전체를 정밀 과금하지 않는다. v1은 단순 멤버십 경계에 집중한다.

## Recommended V1 Payment Approach

v1 권장안은 Toss Payments 자동결제(빌링키) 기반 웹/PWA 구독이다.

이유:

- 한국 사용자의 카드 결제 UX와 PG 심사 흐름에 맞다.
- 네이티브 앱 없이 웹/PWA에서 먼저 유료 전환을 검증할 수 있다.
- buddy-note v1의 핵심은 결제 수단 다양성이 아니라 프리미엄 가치 검증이다.
- 단일 PG로 시작하면 결제 상태, webhook, 고객 문의 대응 정책을 단순하게 유지할 수 있다.

대안 판단:

- PortOne: PG 추상화가 필요하거나 향후 PG 교체 가능성을 크게 보고 싶을 때
- Apple In-App Purchase / Google Play Billing: 네이티브 앱에서 디지털 구독을 직접 판매할 때
- Stripe: 해외 결제 또는 글로벌 확장 검증이 v1 우선순위가 될 때

v1에서는 제공자를 하나로 확정하고, 다른 PG로의 추상화는 실제 필요가 생기기 전까지 제품 스펙 범위에 넣지 않는다.

## Membership And Package Boundaries

v1 패키지는 `무료`와 `멤버십` 두 단계로 시작한다. 일회성 크레딧, 개별 테마 구매, 가족별 좌석 과금은 v1에서 제외한다.

### Free

- 첫 pet 생성
- 기본 AI 일기 생성
- 기본 타임라인 열람
- 기본 공유 이미지
- 제한된 memory summary 보존

무료는 “내 강아지가 나답게 말한다”는 핵심 가치를 이해시키는 구간이다. 프리미엄 화면을 막기 전에 결과물을 먼저 경험하게 한다.

### Membership

- 프리미엄 테마
- 가족 초대와 가족 알림
- 월간 하이라이트 앨범
- 고급 공유 이미지 템플릿
- 긴 memory summary 보존
- 향후 1년 다큐멘터리/연간 회고 기능의 우선 접근

멤버십은 “더 많이 생성”보다 “더 오래 기억하고 더 예쁘게 공유하고 가족이 함께 보는” 가치에 묶는다.

## Trial Policy

기본값은 카드 없는 7일 무료 체험이다.

기본 정책:

- 시작 시점: 첫 pet 생성 완료 후 시작
- 체험 기간: 7일
- 카드 등록: 체험 시작에는 요구하지 않음
- 중복 체험: 같은 user 기준 1회
- 체험 중 권한: 멤버십과 동일한 프리미엄 entitlement 부여
- 체험 종료: 유료 전환이 없으면 프리미엄 entitlement 제거
- 종료 안내: 종료 2일 전, 종료 당일, 종료 후 첫 프리미엄 진입 시점에 안내

열린 질문:

- 첫 기록 작성 후 체험 시작이 전환율과 제품 이해도에 더 유리한지
- 카카오 로그인, 이메일 로그인, 결제수단 기준을 조합해 중복 체험을 막아야 하는지
- 초기 출시에서 7일보다 14일이 월간 하이라이트 가치 전달에 유리한지
- 종료 직후 짧은 grace period를 줄지, 결제 실패 상태에만 grace period를 둘지

## Billing States And User-Facing Behavior

사용자는 모든 상태에서 “지금 무엇을 할 수 있고, 다음에 무엇이 일어나는지”를 이해해야 한다.

| 상태 | 사용자 행동 |
|---|---|
| 무료 | 기본 기능 사용 가능. 프리미엄 진입 시 체험 또는 구독 제안 |
| 무료 체험 중 | 프리미엄 사용 가능. 남은 체험 기간과 전환 CTA 표시 |
| 유료 구독 중 | 프리미엄 사용 가능. 다음 결제일, 결제수단, 해지 진입점 표시 |
| 해지 예약됨 | 결제 기간 종료일까지 프리미엄 유지. 재개 CTA 표시 |
| 결제 실패 grace period | 프리미엄 임시 유지. 결제수단 수정 CTA와 종료 예정일 표시 |
| 구독 종료됨 | 프리미엄 잠금. 기존 일기와 기본 기록은 보존 및 열람 가능 |
| 환불됨 | 환불 정책에 맞춰 프리미엄 제거 또는 기간 종료 처리 |

필수 billing surface:

- 가격/멤버십 비교
- 결제 시작
- 현재 멤버십 상태
- 다음 결제일
- 결제수단 상태
- 해지, 재개, 결제수단 수정
- 결제 실패 복구 안내

프리미엄이 종료되어도 사용자가 만든 pet, log, diary는 삭제하지 않는다. 단, 프리미엄 테마 적용, 가족 기능, 확장 memory summary, 프리미엄 하이라이트 생성 같은 entitlement는 잠근다.

## Entitlement And Server Trust Principles

유료 기능 접근은 UI 표시가 아니라 서버 판정이 기준이다. 클라이언트는 현재 상태를 보여줄 수 있지만, 최종 권한 판단은 서버가 가진 subscription, trial, payment event, entitlement 상태를 기준으로 한다.

원칙:

- 서버는 결제 제공자의 결과와 내부 상태를 함께 확인한 뒤 entitlement를 부여한다.
- webhook, 결제 성공 redirect, 운영자 조정은 같은 상태 모델로 수렴해야 한다.
- 프리미엄 기능을 생성하거나 공개 URL에 반영할 때도 entitlement를 다시 확인한다.
- 클라이언트에 저장된 멤버십 플래그, URL 파라미터, localStorage는 권한 근거로 쓰지 않는다.
- 결제 상태가 불명확하면 유료 권한을 새로 부여하지 않는다. 이미 부여된 grace period는 정책 범위 안에서만 유지한다.

필요 entitlement:

- 프리미엄 테마 사용
- 가족 초대와 가족 알림
- 월간 하이라이트 생성
- 프리미엄 공유 이미지 템플릿 사용
- 확장 memory summary 보존
- 향후 연간 회고/다큐멘터리 접근

## Payment Lifecycle

### Payment Start

사용자는 멤버십 비교 화면 또는 프리미엄 기능 잠금 화면에서 결제를 시작한다. 결제 전 가격, 결제 주기, 체험 여부, 해지 방법, 약관 링크가 명확해야 한다.

### Payment Method Registration

사용자는 결제창에서 결제수단을 등록한다. 등록 성공 후에만 구독 상태를 활성화한다. 실패 시에는 실패 원인과 복구 액션을 보여주되, 프리미엄 entitlement를 부여하지 않는다.

### Recurring Charge

정기 결제 성공 시 다음 결제일까지 멤버십을 유지한다. 실패 시 즉시 사용자에게 결제수단 수정을 안내하고, 기본 grace period 동안만 프리미엄을 유지한다.

기본 grace period는 3일로 둔다. 이 기간은 결제 실패 복구를 위한 정책이며, 무료 체험 종료에는 기본 적용하지 않는다.

### Cancellation And Refund

해지 예약 상태에서는 이미 결제된 기간 종료일까지 프리미엄을 유지한다. 환불은 유료서비스 이용약관과 PG 정책을 따른다. 환불 완료 후 entitlement 제거 시점은 환불 유형에 따라 즉시 또는 기간 종료로 나뉠 수 있다.

### Webhook

결제 성공, 실패, 해지, 환불 이벤트는 webhook을 기준 사실로 남긴다. 동일 이벤트가 여러 번 와도 사용자 상태가 중복 변경되지 않아야 하며, webhook 처리 실패는 운영자가 확인할 수 있어야 한다.

## Data Requirements

구체 schema는 구현 plan에서 확정한다. 제품 스펙 기준 v1에서 필요한 데이터 범위는 아래와 같다.

- 사용자별 현재 멤버십 상태
- plan/package 식별 정보
- 무료 체험 시작/종료/사용 여부
- 현재 결제 기간과 다음 결제 예정일
- 결제수단 등록 상태
- 결제 성공/실패/환불 이력
- webhook 원본 이벤트와 처리 상태
- entitlement 계산 결과와 마지막 갱신 시점
- 운영자 조정 이력

DB 컬럼은 기존 컨벤션대로 `user_id`를 사용한다. migration은 forward-only 원칙을 따른다.

## Operations Requirements

운영자는 최소한 아래 질문에 답할 수 있어야 한다.

- 이 사용자는 현재 무료, 체험, 유료, 실패, 해지 중 어디에 있는가
- 다음 결제일 또는 프리미엄 종료일은 언제인가
- 마지막 결제 실패 원인은 무엇인가
- webhook은 정상 수신되고 처리되었는가
- 환불 또는 수동 조정이 있었는가
- 특정 사용자의 entitlement가 왜 열렸거나 잠겼는가

v1에서는 전용 관리자 UI보다 SQL/admin query로 확인 가능한 상태를 먼저 목표로 한다. 고객 응대에 필요한 최소 정보는 사용자 이메일, user id, subscription status, trial end, current period end, last payment status, webhook event history다.

## Legal And Policy Requirements

결제 시작 전 아래 문서 링크가 준비되어야 한다.

- 이용약관
- 개인정보처리방침
- 유료서비스 이용약관
- 환불/해지 정책

제품 요구사항:

- 가격, 결제 주기, 무료 체험 조건, 자동 결제 전환 여부를 결제 전 명확히 표시한다.
- 해지 방법과 해지 후 이용 가능 기간을 사용자가 찾을 수 있어야 한다.
- 환불 정책은 결제 페이지와 고객 지원 문맥에서 접근 가능해야 한다.
- PG 심사에 필요한 사업자 정보, 고객센터 연락처, 약관 URL을 준비한다.
- 개인정보와 결제 정보는 최소 수집 원칙을 따른다. 카드 원문 정보는 buddy-note가 직접 보관하지 않는다.
- 가족 기능은 초대받은 가족의 개인정보 노출 범위와 탈퇴/초대 해제 동작을 명확히 해야 한다.

## Success Metrics

- 무료 체험 시작률
- 무료 체험 후 유료 전환율
- 프리미엄 잠금 화면에서 체험/결제로 이어지는 비율
- 결제 실패율
- 결제 실패 후 복구율
- 해지율
- 환불률
- 프리미엄 기능 사용률
- 멤버십 사용자의 월간 기록 생성 유지율
- 월간 하이라이트 생성률

## Launch Readiness Criteria

- v1 결제 제공자, 가격, 결제 주기, 체험 기간이 확정되어 있다.
- 무료/멤버십 기능 경계가 온보딩, 기록, 공유, 가족, 하이라이트 화면에서 일관되게 설명된다.
- 결제 성공, 실패, 해지, 환불, webhook 중복 수신 상태를 운영자가 확인할 수 있다.
- 프리미엄 entitlement가 서버 기준으로만 부여되고, 종료/실패 상태에서 잠기는지 검증되어 있다.
- 결제 실패 grace period와 무료 체험 종료 정책이 고객 안내 문구까지 확정되어 있다.
- 이용약관, 개인정보처리방침, 유료서비스 이용약관, 환불/해지 정책 URL이 준비되어 있다.
- PG 심사에 필요한 사업자 정보와 고객센터 정보가 준비되어 있다.
- 고객 문의 대응을 위한 최소 운영 runbook이 준비되어 있다.

## Open Questions

- 무료 체험 시작 시점을 첫 pet 생성으로 확정할지, 첫 기록 작성으로 늦출지
- 7일 체험으로 충분한지, 월간 하이라이트 가치를 위해 14일이 필요한지
- 동일 사용자의 중복 체험 방지를 user 기준으로 충분히 볼지, 결제수단 기준까지 볼지
- 첫 번째 강한 유료 전환 트리거를 프리미엄 테마, 가족 초대, 월간 하이라이트 중 무엇으로 둘지
- 네이티브 앱 출시 시점에 웹 구독과 인앱결제를 병행할지, 플랫폼별 정책을 분리할지
